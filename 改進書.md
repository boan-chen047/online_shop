# 改進書

本文件整理本 session 到目前為止的工作流程、使用過的 skill、實際用法、發生過的錯誤，以及使用者中途提出並已納入的修正偏好。

## 一、使用過的 Skill

### thinking-gate

原本用途：

- 在非簡單任務前釐清目標、限制、假設、最小可行做法與驗證方式。
- 避免還沒理解問題就直接修改。

本次實際用法：

- 用在商品列表設計調整、Supabase schema 設計、Auth 改造等較大範圍任務。
- 多數情況使用者已明確要求「直接執行」，因此沒有停下來要求批准，而是做內部收斂後直接實作。

改進：

- 當使用者已連續多次要求直接修正 UI 時，不應過度輸出計畫，應快速理解並實作。
- 但碰到資料庫、Auth、RLS 這類影響較大的部分，仍應簡短說明風險與執行順序。

### stage-transition-check

原本用途：

- 在研究、規劃、實作、驗證、回報之間切換時防止任務漂移。

本次實際用法：

- 用在從 UI 研究轉實作、從實作轉驗證、從 Supabase 設計轉 migration 等階段。

改進：

- 對 UI 微調任務，階段檢查應更短。
- 對資料庫任務，階段檢查有必要，因為錯誤會影響遠端資料庫。

### design-review

原本用途：

- 檢查 UX、視覺比例、資訊層級、可點擊性、響應式體驗。

本次實際用法：

- 用在商品列表頁、購物車頁、商品詳情頁、HomeView、登入頁。
- 主要判斷問題包含：
  - 商品圖片與文字比例不平衡。
  - 商品名稱與價格距離過大。
  - 桌機與手機商品卡比例應分開。
  - 登入頁輸入區被大型左側區塊壓到下方。
  - UserFile 未登入中介頁多餘。

使用者修正後形成的偏好：

- 桌機商品 grid 優先一排 4 個。
- 桌機商品卡使用「上圖下文」。
- 桌機商品圖片接近方形，但比完整正方形少約 15% 面積。
- 手機版可以使用左圖右文 compact card。
- 商品名與價格要視覺成組，不要隔太遠。
- 未實作的評分、review、wishlist、share 不要提前出現在 UI。

### modern-web-dev

原本用途：

- 修改現代前端應用，處理 Vue/Vite/Tailwind UI、狀態、驗證、瀏覽器檢查。

本次實際用法：

- 修改多個 Vue component：
  - `ProductList.vue`
  - `ProductDetails.vue`
  - `ShoppingCart.vue`
  - `HomeView.vue`
  - `Login.vue`
  - `UserFile.vue`
  - `NavBar.vue`
  - `Footer.vue`
- 每次主要 UI 修改後執行 `pnpm build`。
- 依使用者全域偏好，使用 Chrome 做測試。

改進：

- 開 Chrome 前應先確認 dev server 是否正在跑。
- 偏好使用 Chrome 做視覺確認，但不應每次測試都開新視窗。
- Vue/Vite 有 hot module replacement，UI 修改後應優先復用既有頁面或 Chrome session。
- 若 `127.0.0.1:5173` 無法連線，應先啟動 Vite，不應讓使用者看到 connection refused。

本次實際用法：

  - `supabase/migrations/202605120004_rename_profiles_to_user_profile.sql`

本次資料表設計：

- `user_profile`

- `products`
- `product_images`
- `product_specs`
- `inventory`
- `cart_items`

### workflow-improvement-loop

原本用途：

- 紀錄執行過程中的錯誤、摩擦、環境問題、使用者修正，並在事後形成改進建議。

本次實際用法：

- 紀錄了 Playwright 不存在導致無法自動截圖。

- 紀錄了 Supabase grants 遺漏造成 profile 無法寫入。

- Supabase schema 建完後應檢查：
  - table 是否存在
  - RLS 是否開啟
  - grant 是否存在
  - 前端 publishable key 是否有足夠 Data API 權限

原本用途：

## 二、出現過的錯誤與處理

### 1. `pnpm.ps1` Access Denied

現象：

```txt
Access to the path 'C:\Users\User\AppData\Roaming\npm\pnpm.ps1' is denied.
```

原因：

- Windows 下直接執行 `.ps1` 可能被權限或執行政策擋住。

修正：

- 改用 PowerShell 包一層：
  ```powershell
  powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "& 'C:\Users\User\AppData\Roaming\npm\pnpm.ps1' build"
  ```

### 2. Chrome 開啟後顯示 connection refused

現象：

```txt
127.0.0.1 refused connection
ERR_CONNECTION_REFUSED
```

原因：

- Vite dev server 沒在跑，但我直接開了 Chrome。

修正：

- 先啟動 dev server。
- 再用 `Invoke-WebRequest http://127.0.0.1:5173/login` 確認 200。
- 最後再開 Chrome。

改進：

- 任何 Chrome 測試前，都應先檢查 dev server 狀態。
- 若 dev server 已在跑，優先刷新或復用既有 Chrome 頁面，不要每次開新視窗。

### 3. Node REPL 沒有 Playwright

現象：

```txt
Module not found: playwright
```

原因：

- Node REPL 環境沒有安裝 Playwright。

修正：

- 改用實際 Chrome 開頁面檢查。

改進：

- 若需要自動視覺回歸測試，應後續正式加入 Playwright。

### 4. Supabase `profiles` 寫不進去

現象：

- Google 登入成功。
- 前端顯示使用者姓名。
- 但 Supabase `profiles` table 空白。
- 前端出現泛用錯誤：
  ```txt
  Unable to complete the authentication request. Please try again.
  ```

原因：

- 使用者關閉了 Supabase Automatically expose new tables，這是安全正確的。
- 但 migration 忘記 grant `profiles` 給 authenticated Data API role。
- 既有 Google user 是 trigger 建好之前登入的，也需要 backfill。

修正：

- 新增：
  - `202605120003_grants_and_profile_backfill.sql`
- 補上：
  - `grant select, insert, update on public.profiles to authenticated`
  - 回填 `auth.users` 到 `profiles`
- 後來再改名成 `user_profile`。

改進：

- RLS policy 不等於 Data API 權限。
- 只要關閉 auto expose，就要手動 grant。

### 5. `create_order_from_cart()` 不存在

現象：

```txt
ERROR: 42883: function public.create_order_from_cart() does not exist
```

原因：

- 第三份 migration 直接 grant function，但 DB 內當時沒有該 function。

修正：

- 將 grant 包進 `do $$ begin if exists (...) then grant ... end if; end; $$;`

改進：

- grant function 時應考慮 function 尚未建立或前一份 SQL 未完整執行的情況。

### 6. SQL Editor 顯示 destructive warning

現象：

- Supabase SQL Editor 顯示 query 有 destructive operations。

原因：

- SQL 裡有：
  ```sql
  drop policy if exists ...
  drop trigger if exists ...
  ```

說明：

- 此處不是刪除資料表或資料，是為了 migration 可重複執行。

改進：

- 回覆時應明確說明 drop 的目標是 policy/trigger，不是資料。

### 7. Google OAuth client secret 暴露在截圖

現象：

- 使用者截圖中顯示 Google OAuth client secret。

風險：

- Client secret 屬於敏感資訊，不應公開。

建議：

- 到 Google Cloud Console rotate client secret。
- 更新 Supabase Google provider 內的 secret。

- 當購物車、訂單、Auth 串接完成後，應補最小可靠的回歸測試或 E2E smoke test。

## 七、設計自動化決策

目前設計相關 skill 已存在，但角色偏審查，不足以保證使用者不給提示時也能自動產出好看的 UI。

決策：

- 全域 agent 放輕量原則：任何 user-facing UI 改動，都要主動做一次設計品質 pass。
- `design-review` skill 放具體流程：產品類型判斷、資訊層級、版面密度、spacing、typography、responsive、states、accessibility。
- `online_shop/AGENTS.md` 放專案偏好：商品卡、登入頁、UserFile、Supabase 決策。
- 全域保留主要內容左右留白與 typography hierarchy 規則；商品卡欄數、比例等細節留在專案層。

這樣可以避免兩個問題：

- 只改全域 agent：規則太抽象，設計落地仍不穩。
- 只改 skill：若 skill 沒被觸發，設計品質仍可能掉下來。
